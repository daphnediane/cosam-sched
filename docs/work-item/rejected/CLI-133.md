# CLI-133: cosam-layout output as file paths, not directories

## Summary

Redesign `cosam-layout` to accept `--output <file>` (explicit file path) rather
than `--output-dir <dir>`, mirroring the old `dump_flyers` / `schedule_html`
pattern where each job spec points to a specific output file or directory.

## Status

Rejected

## Resolution

Superseded by **CLI-139**. The standalone `cosam-layout` binary has been
removed, so redesigning its `--output` handling no longer applies. The goal —
defining a layout job and pointing it at an explicit output file — is now met by
`cosam-convert`: `--layout.<key>=<value>` flags build a single job and
`--export-layout <file>` writes it to that path (verbatim when the path has an
extension). The `--format` shorthand described below maps onto
`--layout.import=<preset>` plus the `content`/`split`/`paper` keys, and the
broken standalone widget-JSON loader is moot because `cosam-convert` builds
`ScheduleData` in-process via `from_schedule`.

## Priority

Medium

## Known Issues / Testing

⚠️ **`cosam-layout` is currently broken for standalone use.** Its widget-JSON
loader (`ScheduleData::load` / `from_json`) rejects the JSON that
`cosam-convert --export` produces — e.g. `JSON parse error: missing field
\`uid\`` on presenters — so the standalone CLI cannot round-trip exported
schedules. The native `.schedule` binary input path still works.

To test the layout engine, **use `cosam-convert --export-layout <dir>`** instead
of invoking `cosam-layout` directly. That path builds `ScheduleData` in-process
via `ScheduleData::from_schedule` (no JSON round-trip) and runs the jobs from
`config/layout.toml` / the embedded defaults. Fixing the standalone JSON loader
(aligning the `model.rs` deserialization defaults with the exported widget JSON)
is part of this work item.

## Description

The current `cosam-layout` global `--output-dir` flag writes all job outputs
into a single directory, deriving filenames from format-generated stems.
The old `schedule_html` tool (see `schedule-to-html/dump_flyers`) accepted
`--output <path>` per job, where the path was either:

- A file path (e.g. `output/flyers/poster17x11_grid.html`) → single output
- A directory path (trailing `/` or existing dir) → one file per day/room/guest

`cosam-layout` should adopt the same pattern:

- Remove global `--output-dir`; replace with per-job `--output <path>`
- If the path looks like a directory (trailing `/`, or is an existing dir),
  write multiple files into it using generated stems
- If the path looks like a file, use it as the output path; warn if the format
  produces multiple files and only the first is written (or error out)
- `--typ` / `--no-compile` flags remain global
- `compile_typst` already accepts explicit `typ_path` and `pdf_path` after
  the Phase 4 refactor — just need to wire paths correctly

## Groundwork Completed (pre-requisites done)

- Format modules (`schedule`, `descriptions`, `workshops_listing`, `room_signs`,
  `guest_postcards`) now return `(split_qualifier, typ_source)` pairs instead of
  full filename stems. Name decisions belong to the caller.
- `cosam-layout` has `--stem <base>` per job; falls back to `{format}-{paper}`
  default slug when omitted. `run_job` combines `base_stem + "-" + qualifier`.
- `cosam-convert` uses a `LayoutOutputJob { config, stem }` struct with
  hardcoded stems (e.g. `"desc-tabloid"`, `"desc-letter"`); outputs go flat in
  `layout_dir/` instead of per-paper subdirs.
- `--output` single-file shortcut is wired: if `output_override` has an
  extension and there is only one output, it is used as the direct PDF path.

## Acceptance Criteria

- `cosam-layout --input schedule.json -- --output output/grid-friday.pdf --format schedule --paper tabloid` writes a single PDF
- `cosam-layout --input schedule.json -- --output output/rooms/ --format room-signs --paper tabloid` writes one PDF per room per day into `output/rooms/`
- `cosam-layout --input schedule.json -- --output output/rooms/ --grids --descriptions --paper tabloid --split-day --split-rooms -- --output output/grid-friday.pdf --select-day friday --descriptions --paper tabloid` writes one PDF per room per day into `output/rooms/` and a single PDF to `output/grid-friday.pdf` equivalent to the both commands.
- `--format` should just be a short hand for a selection of options describing what to put on the page, and what data to include, and how to split the output in the case of multiple files.
- Help text clearly describes when to use file vs. directory output
